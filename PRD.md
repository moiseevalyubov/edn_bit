# PRD: Приложение интеграции MAX-бот с Открытыми линиями Битрикс24

## 1. Overview

### Что это
Приложение для маркетплейса Битрикс24, которое подключает канал MAX-бот к Открытым линиям. Позволяет операторам принимать и отправлять сообщения (текст, изображения, файлы, видео, аудио) абонентам MAX-бот прямо из интерфейса Битрикс24.

### Зачем это нужно
Компании, которые используют Битрикс24 как CRM и MAX-бот как канал коммуникации, вынуждены работать в двух разных интерфейсах. Это приводит к потере обращений, медленным ответам и невозможности инициировать диалоги с клиентами из CRM.

### Архитектура
- Уже существует сервер, подключённый к MAX-бот, с API для отправки сообщений и вебхуком для получения входящих сообщений
- Приложение Битрикс24 выступает посредником: принимает входящие с сервера и передаёт в Открытые линии, а исходящие от операторов — обратно на сервер

---

## 2. Целевой сегмент

### Кто эти люди
Малый и средний бизнес с активными продажами через мессенджеры. Менеджеры по продажам и руководители отделов продаж в компаниях, которые уже используют Битрикс24 на платном тарифе и MAX-бот как канал коммуникации с клиентами.

### Big Job
Не терять ни одного обращения клиента и конвертировать больше диалогов в продажи, чтобы растить выручку без увеличения рекламного бюджета.

### Core Jobs

**Core Job 1: Быстрый ответ клиенту из CRM**
- **Когда** клиент пишет в MAX-бот с вопросом о товаре/услуге, **хочу** быстро ответить ему прямо из Открытых линий Битрикс24, не переключаясь между окнами
- **Критерии успеха:** сообщение от клиента появляется в Открытых линиях; оператор отвечает в том же окне; ответ доставляется клиенту в MAX-бот

**Core Job 2: Инициация диалога с клиентом**
- **Когда** клиент давно не покупал, **хочу** написать ему первым в мессенджер через Открытые линии с персональным предложением
- **Критерии успеха:** возможность начать диалог с абонентом, который ранее уже писал в MAX-бот

## 3. Scope MVP

### Что входит в MVP

#### 4.1. Подключение канала MAX-бот
- Веб-форма с 3 текстовыми полями:
  - **Название канала** — произвольное имя для идентификации
  - **API-ключ** — ключ для обращения к серверу MAX-бот
  - **Sender** — идентификатор приложения MAX-бот
- Один портал Битрикс24 может подключить несколько каналов MAX-бот
- Валидация API-ключа при подключении не требуется — просто сохраняем данные
- Фиксируется дата/время каждого подключения и отключения канала
- Во время сохранения канала на бэкенде показывается индикатор загрузки (loading/progress)
- После успешного сохранения канала сервер запрашивает у Битрикс24 URL для приёма вебхуков с входящими сообщениями Открытых линий и возвращает его в ответе; приложение показывает этот URL пользователю, чтобы он мог скопировать его и указать на сервере MAX-бот
- Канал можно отключить после сохранения: при отключении канал не удаляется из БД, а помечается как отключённый; фиксируется дата/время отключения
- При любых ошибках (сохранение, отключение) показывается сообщение об ошибке от бэкенда как есть, без изменений

#### 4.2. Приём входящих сообщений
- Приложение принимает вебхуки от сервера MAX-бот с входящими текстовыми сообщениями
- Входящее сообщение создаёт диалог в Открытых линиях Битрикс24 (или добавляется в существующий)
- Если канал из вебхука не найден в БД — сообщение логируется и игнорируется
- Эндпоинт `/incoming` принимает методы **POST**, **GET** и **HEAD**: edna проверяет доступность URL двумя запросами (GET + HEAD) перед сохранением — оба должны вернуть 200 OK

**Формат вебхук-запроса с входящим сообщением:**

Структура запроса описана в манифесте: [MAX Bot Incoming - LLM Manifest.md](MAX Bot Incoming - LLM Manifest.md)

#### 4.3. Отправка исходящих сообщений
- Оператор пишет ответ в Открытых линиях — сообщение передаётся на сервер MAX-бот через API
- Поддерживаются типы контента: **TEXT**, **IMAGE**, **VIDEO**, **AUDIO**, **DOCUMENT**
- При отправке файла (изображение, документ и т.д.) сервер определяет тип по MIME-типу и расширению файла, после чего передаёт файл в edna через прокси-эндпоинт `/file/{filename}` — это необходимо, потому что edna требует URL с расширением файла в пути
- Если сервер MAX-бот недоступен или вернул ошибку — оператору показывается ошибка в интерфейсе Открытых линий

**Формат API-запроса отправки сообщения:**

Структура запроса описана в манифесте: [MAX Bot send message - LLM Manifest.md](MAX Bot send message - LLM Manifest.md)

#### Соответствие параметров между API отправки и вебхуком входящих

Одни и те же сущности имеют разные имена полей в двух API:

| Сущность | API отправки (4.3) | Вебхук входящих (4.2) |
|---|---|---|
| Идентификатор канала | `sender` | `subject` |
| Идентификатор абонента в MAX | `maxId` | `subscriber.identifier` |

#### 4.3.1. Выбор открытой линии

- В настройках приложения отображается раздел **«Открытая линия Битрикс24»**
- Администратор выбирает открытую линию из выпадающего списка — список загружается из Битрикс24 в реальном времени
- После выбора: приложение активирует коннектор `max_bot` на выбранной линии и сохраняет её ID
- Если линия выбрана — отображается её название и кнопка «Изменить»
- Если линии в Битрикс24 есть, но ни одна не выбрана — показывается предупреждение с кнопкой «Выбрать линию»
- Если в Битрикс24 нет ни одной открытой линии — показывается кнопка **«Создать линию»**: приложение автоматически создаёт линию с названием «MAX Bot» через API Битрикс24 и активирует на ней коннектор
- Если линия не выбрана — входящие сообщения не маршрутизируются
- Администратор может изменить линию в любой момент через ту же форму
- **Не делаем:** автовыбор первой доступной линии при установке — опасно, может перезаписать настройки существующих интеграций на этой линии

#### 4.3.2. Установка приложения

- При установке Битрикс24 передаёт параметры авторизации: `AUTH_ID`, `REFRESH_ID`, `MEMBER_ID`, `DOMAIN`, `PROTOCOL`
- Приложение строит адрес REST API портала (`client_endpoint`) из параметров `DOMAIN` и `PROTOCOL`, если он не передан напрямую
- После сохранения данных портала — регистрирует коннектор `max_bot` и привязывает события (`OnImConnectorMessageAdd`, `OnImConnectorDialogStart`, `OnImConnectorDialogFinish`)
- Если регистрация коннектора или привязка событий упала при установке — это не блокирует установку, но устраняется автоматически при следующем открытии страницы настроек
- При открытии страницы настроек приложение автоматически: восстанавливает `client_endpoint` если он пустой, перерегистрирует коннектор и события — это обеспечивает корректную работу даже если что-то пошло не так при установке
- Страница настроек (`/settings`) принимает как GET-, так и POST-запросы: Битрикс24 открывает `PLACEMENT_HANDLER` через POST, браузерный переход — через GET
- URL обработчика событий формируется из переменной окружения `APP_BASE_URL` — она должна быть задана явно при деплое; если не задана, при старте сервера выводится ошибка в лог и события работать не будут
- Битрикс24 доставляет события (`OnImConnectorMessageAdd` и др.) на эндпоинт `/handler` в формате `application/x-www-form-urlencoded` с PHP-style вложенными ключами (например, `data[MESSAGES][0][chat][id]`), а не JSON. Обработчик парсит такой формат в нужную структуру. Если в теле запроса есть поле `event` — это событие Битрикс24, обрабатывается как событие. Если поля `event` нет — это открытие страницы через iframe, выполняется редирект на `/settings`

#### 4.4. Мультитенантность
- Одно приложение обслуживает множество порталов Битрикс24
- Каждый портал подключает и настраивает свои каналы независимо

#### 4.5. Хранение данных

Приложение должно хранить следующие данные (без ограничений по сроку и объёму). Конкретное решение для хранения выбирается на этапе разработки — возможно использование встроенных механизмов Битрикс24 или внешней БД:

- Информация о каждом портале Битрикс24, подключившем приложение
- Информация о каждом подключённом канале для каждого портала (включая дату/время подключения и отключения)
- Все сообщения с атрибутами:
  - Текст сообщения
  - Время отправки/получения
  - Направление (входящее в Битрикс / исходящее из Битрикс)
  - Контент (тип содержимого)
  - Другие атрибуты сообщения (идентификатор абонента, идентификатор диалога и т.д.)

#### 4.6. Логирование
- Логирование всех входящих и исходящих HTTP-запросов для отладки
- При обработке исходящего сообщения (ответ оператора → MAX Bot) логируется каждый шаг: полученные данные, причина пропуска если сообщение не обработано, факт отправки в edna и ответ edna
- При регистрации событий в Битрикс24 логируется URL обработчика, который передаётся Битрикс24
- При старте сервера логируется значение `APP_BASE_URL` — критически важная переменная для работы событий

#### 4.7. Производительность
- Пропускная способность: не менее 50 сообщений в секунду (входящие + исходящие суммарно)

### Что НЕ входит в MVP
- Валидация API-ключа при подключении канала
- Лимиты на хранение данных

## 5. Поведение в edge cases

| Ситуация | Поведение |
|---|---|
| Оператор закрыл диалог, абонент пишет снова | Поведение определяется стандартной логикой Открытых линий Битрикс24 |
| Сервер MAX-бот недоступен при отправке | Показать оператору ошибку в интерфейсе Открытых линий |
| Несколько каналов MAX-бот на одном портале | Поддерживается, без ограничений на количество |
| Клиент удалил (деинсталлировал) приложение | Данные о каналах и сообщениях сохраняются в БД. Фиксируется дата/время отключения |
| Невалидный API-ключ | Не проверяется при подключении. Ошибки проявятся при отправке сообщений |
| Входящий вебхук для неизвестного канала | Логируется и игнорируется |
| Хранение сообщений | Без ограничений по сроку и объёму |
