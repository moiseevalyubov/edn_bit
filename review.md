# Review: замечания по коду

## ✅ ИСПРАВЛЕНО — 1. Синхронные HTTP-вызовы блокируют event loop

**Файлы:** `app/services/maxbot.py:6`, `app/services/bitrix.py:15`

`httpx.post` вызывается синхронно внутри async FastAPI-хендлеров. При медленном ответе edna или Bitrix (300–500ms) event loop блокируется, и реальная пропускная способность составит 2–3 сообщения в секунду вместо требуемых 50.

**Решение:** заменить синхронный `httpx.post` на `async with httpx.AsyncClient` с `await`.

---

## 2. Ошибка отправки в MAX Bot не сообщается оператору

**Файл:** `app/routers/handler.py:202`

При сбое отправки сообщения в edna ошибка только логируется. В Bitrix24 не отправляется `imconnector.send.status.undelivered`, поэтому оператор не видит, что сообщение не доставлено. Нарушает требование PRD раздел 4.3.

**Решение:** при ошибке отправки в edna вызывать `imconnector.send.status.undelivered` с идентификаторами сообщения.

---

## 3. SQLite по умолчанию — риск при деплое с несколькими workers

**Файл:** `app/config.py:4`

`database_url` по умолчанию указывает на SQLite. SQLite не поддерживает одновременные записи от нескольких Uvicorn workers. Если при деплое не задать `DATABASE_URL`, приложение молча запустится на SQLite и будет давать ошибки при нагрузке.

**Решение:** либо сделать `DATABASE_URL` обязательной переменной (без дефолта), либо явно документировать риск и выводить предупреждение при старте если используется SQLite.

---

## ✅ ИСПРАВЛЕНО — 4. Манифесты MAX Bot не описывают файловый API

**Файлы:** `MAX Bot Incoming - LLM Manifest.md`, `MAX Bot send message - LLM Manifest.md`

Входящий манифест показывает `attachment: null` в примере — вложения технически существуют, но не задокументированы. Исходящий манифест описывает только `content.type: TEXT`, эндпоинта для отправки файлов нет. Без обновлённых манифестов нельзя планировать реализацию файлового обмена.

**Решение:** получить от edna обновлённые манифесты с описанием файлового API (входящий вебхук с вложением и исходящий запрос с файлом).

---

## ✅ ИСПРАВЛЕНО — 5. Синхронный httpx станет блокером при передаче файлов

**Связано с замечанием №1.**

Файл 5–10 MB скачивается и загружается 3–10 секунд. Синхронный `httpx.post` (замечание №1) при текстовых сообщениях терпим, но при файлах заблокирует event loop на всё время передачи. Замечание №1 должно быть исправлено **до** реализации файлового обмена.

**Решение:** исправить замечание №1 (async httpx), затем добавлять файловый обмен.

---

## 6. Нет очереди задач — файловый обмен вызовет таймауты

При добавлении файлового обмена обработка вебхука будет включать: скачать файл → загрузить в Bitrix/edna. На медленной сети это может занять десятки секунд и превысить таймаут запроса. Текущая архитектура обрабатывает всё синхронно в рамках HTTP-запроса.

**Решение:** для файловых сообщений рассмотреть фоновую обработку — принять вебхук, ответить 200, обработать файл в фоне (например, через `BackgroundTasks` FastAPI для лёгких случаев или Celery/ARQ для надёжности).
